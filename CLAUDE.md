# CLAUDE.md — 위키 스키마 (LLM 운영 규칙)

> 이 파일은 **위키의 운영체제**다. LLM(Claude/Claude Code)이 이 학습 위키를
> 어떻게 컴파일하고 유지·점검하는지 정의한다. Andrej Karpathy의 **LLM Wiki** 모델을 따른다.
> 작업을 시작하기 전에 이 파일을 반드시 먼저 읽는다.

---

## 0. 한 줄 요약

학생(중3)이 `sources/{과목}/`에 **문제·풀이 원본**을 넣으면, LLM이 그것을 읽어
`wiki/{과목}/`에 **구조화된 지식 페이지로 컴파일**하고, 교차참조·약점 집계·정합성 점검을 유지한다.
**RAG로 매번 재검색하지 않는다. 한 번 컴파일해서 영구히 유지한다.**

> **멀티 과목:** 이 위키는 **과목별 네임스페이스**(`{과목}` = `math`, `physics`, …)로 나뉜다.
> 과목마다 독립된 problems·concepts·mistakes·dashboard·exams를 가진다. §1.5 참조.

---

## 1. 3계층 아키텍처

| 계층 | 디렉터리 | 소유자 | 규칙 |
|------|----------|--------|------|
| **Raw Sources** | `sources/` | 학생(사람) | **불변(immutable).** LLM은 읽기만 하고 **절대 수정/삭제하지 않는다.** 진실의 원천. |
| **Wiki** | `wiki/` | **LLM** | LLM이 전적으로 소유. 페이지 생성·갱신·교차참조·집계를 LLM이 수행. |
| **Schema** | `CLAUDE.md` (이 파일) | 사람 | 구조·규칙·워크플로 정의. LLM의 행동 지침. |

> **불변성 원칙:** `sources/` 안의 사진·텍스트는 한 번 들어오면 바꾸지 않는다.
> 내용이 바뀌면 새 source를 추가한다(버전처럼). 위키만 갱신한다.

---

## 1.5 과목 네임스페이스 (멀티 과목)

원본과 위키는 **과목(`{과목}`)별로 격리**된다. 한 과목의 컴파일은 다른 과목 페이지를 건드리지 않는다.

- 현재 과목: **`math`**(수학, 활성), **`physics`**(물리, 준비됨).
- 원본: `sources/{과목}/YYYY/MM/`
- 위키: `wiki/{과목}/{problems,concepts,mistakes,dashboard,exams}/`
- **과목 무관 공유 인프라**(과목 폴더 밖, 한 벌만 유지):
  - `wiki/index.md` — 전 과목 진입점 허브.
  - `wiki/log.md` — 전 과목 통합 append-only 로그(과목명을 줄에 포함).
  - `wiki/templates/` — 모든 과목이 공유하는 페이지 형식.
  - `sources/index.md` — 전 과목 원본 카탈로그(과목별 섹션).

**새 과목 추가 절차:** `sources/{과목}/`, `wiki/{과목}/{problems,concepts,mistakes,dashboard,exams}/`를
만들고 각 폴더에 README, `wiki/{과목}/dashboard/weakness-map.md`를 둔 뒤,
`wiki/index.md`·`sources/index.md`·`mkdocs.yml`에 과목 섹션을 추가한다.

## 2. 디렉터리 지도

```text
math-learning-wiki/            # (리포 이름은 유지; 실제로는 다과목 학습 위키)
├─ CLAUDE.md                 # (이 파일) 스키마
├─ sources/                  # ⬛ RAW (불변) — 학생이 넣는 원본
│  ├─ index.md               #   전 과목 원본 카탈로그 (source-id 목록)
│  └─ {과목}/                 #   math/ · physics/ …
│     └─ YYYY/MM/            #   문제 사진·풀이 사진·문제 텍스트(.md)
├─ wiki/                     # 🟦 LLM이 생성·유지하는 지식 계층
│  ├─ index.md               #   전 과목 진입점 허브
│  ├─ log.md                 #   append-only 통합 활동 로그
│  ├─ templates/             #   (공유) 페이지 생성용 표준 형식
│  └─ {과목}/                 #   math/ · physics/ …
│     ├─ problems/           #   문제 1개 = 컴파일된 페이지 1개
│     ├─ concepts/           #   개념/단원 = 엔티티 페이지 (지식 누적)
│     ├─ mistakes/           #   반복 오답 패턴 = 엔티티 페이지
│     ├─ dashboard/          #   weakness-map / weekly-review (집계 페이지)
│     └─ exams/              #   시험 대비 컴파일 결과
├─ prompts/                  # compile / lint / query / 유사문제 / 시험
├─ docs/                     # workflow, privacy
└─ scripts/                  # (선택) 자동화
```

## 3. source-id 규칙 (원본 식별자)

모든 raw source에는 고유 ID를 부여한다. 위키는 항상 이 ID로 원본을 역참조한다.

- 형식: `YYYY-MM-DD-q{n}` (예: `2026-06-07-q1`)
- 파일 (과목 폴더 안):
  - 문제 사진: `sources/{과목}/YYYY/MM/{source-id}.jpg`
  - 풀이 사진: `sources/{과목}/YYYY/MM/{source-id}-sol.jpg`
  - 문제 텍스트(선택): `sources/{과목}/YYYY/MM/{source-id}.md`
- 전 과목 원본 카탈로그 `sources/index.md`의 해당 과목 섹션에 한 줄 등록.

---

## 4. 페이지 타입과 표준 구조

LLM은 아래 형식을 지켜 페이지를 만든다. 형식 원본은 `wiki/templates/`에 있다.

### 4.1 문제 페이지 `wiki/{과목}/problems/{source-id}.md`
컴파일의 기본 단위. 원본 1개 → 페이지 1개.
- frontmatter: `source`, `unit`, `result(정답/오답)`, `cause_tags`, `created`
- 본문: 문제 / 학생 풀이 재구성 / 정답 여부 / 장점 / 약점 / **오답 원인** / 올바른 풀이 / 개선 액션
- **백링크 필수:** 관련 `concepts/`, 관련 `mistakes/` 패턴 페이지로 링크.

### 4.2 개념 엔티티 페이지 `wiki/{과목}/concepts/{단원}.md`
단원/개념은 **누적되는 엔티티**다. 새 문제가 컴파일될 때마다 갱신.
- 핵심 정의·공식 / 자주 하는 실수(데이터 기반) / 연결된 문제 목록 / 숙련도
- 같은 개념의 새 사실이 들어오면 **덮어쓰지 말고 통합**하고, 모순되면 명시.

### 4.3 오답 패턴 엔티티 페이지 `wiki/{과목}/mistakes/{패턴}.md`
**반복되는 실수 유형**을 하나의 엔티티로 본다 (예: `부호처리실수.md`).
- 해당 패턴의 모든 발생 문제를 링크로 모은다 / 반복 횟수 / 교정 전략.
- 새 발생 시 `occurrences`에 추가하고 `repeat_count`를 올린다.

### 4.4 집계 페이지 `wiki/{과목}/dashboard/`
- `weakness-map.md`: 단원·패턴별 약점 롤업. **compile/lint 시 LLM이 재집계.**
- `weekly-review.md`: 주간 회고. query 워크플로로 생성.

## 5. 핵심 워크플로

### 5.1 COMPILE (새 원본 → 위키 통합) — 가장 중요
트리거: 학생이 `sources/{과목}/`에 새 문제/풀이를 추가. 프롬프트: `prompts/compile.md`.
**먼저 어느 과목인지 확정**하고(폴더 위치로 판단), 그 과목 네임스페이스 안에서만 작업한다.

1. 새 source를 읽는다 (사진 + 텍스트). **원본은 수정하지 않는다.**
2. `sources/index.md`의 해당 과목 섹션에 등록되어 있는지 확인, 없으면 추가.
3. `wiki/{과목}/problems/{source-id}.md` 생성 (4.1 형식).
4. 풀이를 분석: 장점·약점·**오답 원인 태그**(§6) 부여.
5. 관련 **개념 페이지**(`wiki/{과목}/concepts/`) 갱신 (없으면 생성). 자주 하는 실수 표에 통합.
6. 오답이면 관련 **오답 패턴 페이지**(`wiki/{과목}/mistakes/`) 갱신/생성, `repeat_count` 증가.
7. 모든 새 페이지에 **양방향 백링크**를 건다 (같은 과목 내에서).
8. `wiki/{과목}/dashboard/weakness-map.md` 재집계.
9. `wiki/index.md` 허브의 해당 과목 섹션에 새 페이지/상태 반영.
10. `wiki/log.md`에 **append-only** 한 줄 기록 (§7, 과목명 포함).

### 5.2 LINT (정합성·건강 점검)
트리거: 주기적(주 1회) 또는 요청 시. 프롬프트: `prompts/lint.md`.
점검 항목:
- **모순(contradiction):** 같은 개념에 상충하는 서술.
- **고아 페이지(orphan):** 어디서도 링크되지 않은 페이지.
- **끊긴/누락 백링크:** A→B는 있는데 B→A가 없음.
- **오래된 주장(stale):** 이미 극복된 약점이 여전히 "위험"으로 표시.
- **데이터 공백(gap):** 단원은 있는데 컴파일된 문제가 0개 등.
- 발견 사항을 보고하고, 안전한 수정은 수행 후 `log.md`에 기록.

### 5.3 QUERY (위키에 질문)
프롬프트: `prompts/query.md`. **원본을 재검색하지 말고 위키(요약/집계)를 읽어** 답한다.
규모가 작으므로(§ No-RAG) `wiki/index.md`로 탐색 후 필요한 페이지만 읽는다.

### 5.4 보조 워크플로
- `prompts/generate-similar-problems.md`: 약점 패턴 기반 유사 문제 생성.
- `prompts/exam-prep.md`: 시험 범위로 약점·오답을 필터링해 대비 자료 컴파일.

## 6. 오답 원인 태그 (고정 어휘)

분석·집계의 일관성을 위해 **반드시 아래 태그만** 사용한다.

| 태그 | 의미 |
|------|------|
| `#계산실수` | 개념은 알지만 연산·부호·이항에서 틀림 |
| `#개념부족` | 정의·공식·정리를 몰라서 틀림 |
| `#문제해석오류` | 문제를 잘못 읽거나 조건을 놓침 |
| `#풀이전략오류` | 접근 방법 자체가 비효율/오류 |
| `#시간부족` | 방법은 알지만 시간 내 못 풂 |
| `#실수누락` | 검산·단위·범위 확인 안 함 |
| `#서술미흡` | 답은 맞지만 풀이 서술 부족 |

한 문제에 여러 태그 가능. 가장 핵심을 **주원인(primary)**으로 표시.

## 7. index.md 와 log.md 유지 규칙

- **`wiki/index.md`** — 내용 중심 카탈로그(진입점). 페이지가 생기거나 의미가 바뀌면 갱신.
- **`wiki/log.md`** — **append-only.** 과거 줄은 절대 수정/삭제하지 않는다. 한 줄 형식:
  ```
  - YYYY-MM-DD HH:MM | <compile|lint|query|exam> | <과목> | <source-id 또는 대상> | <한 줄 요약>
  ```
  > 시각을 모르면 날짜만, 시간은 `--:--`로 둔다(임의 시각 지어내지 않음).
  > `wiki/log.md`는 전 과목 통합 로그다. 과목 칸에 `math`/`physics` 등을 적는다.

---

## 8. No-RAG / 컨텍스트 원칙

이 위키는 소규모(~수백 페이지)이므로 임베딩 RAG를 쓰지 않는다.
`wiki/index.md`(+ 필요한 페이지)만 컨텍스트에 올려 중복·모순을 직접 점검한다.
규모가 매우 커지면 그때 분할/색인 전략을 재검토한다.

---

## 9. 역할 분담

- **학생(사람):** 문제 풀기, 원본 넣기(sourcing), 질문하기, 전략 결정. `sources/`의 최종 권한.
- **LLM:** 요약·교차참조·정리·집계·정합성 점검 등 **모든 grunt work.** `wiki/`의 유지 담당.
- LLM은 `sources/`를 **수정하지 않으며**, 추측이 필요하면 "사진상 ~로 보임"이라고 명시한다.

---

## 10. 작성 규칙

- **언어: 한국어.** 대상이 중3 학생이므로 격려하되 정확하게.
- 약점도 깎아내리지 말고 성장 단서로 표현.
- 수식은 `$...$`(인라인), `$$...$$`(블록).
- 새 페이지는 항상 `wiki/templates/`의 형식을 복사해 시작.

## 11. 개인정보 (필수 준수)

미성년자 데이터다. 실명·학교·연락처 금지, 사진 속 식별정보는 가린다.
자세한 규칙은 [`docs/privacy.md`](docs/privacy.md). 위키 텍스트에 식별정보를 옮겨 적지 않는다.
